### Item 4 — 공개키 배포/로테이션 (ACK)

**Edge 팀 2026-04-21 — ACK, 프로토콜 초안**

현재 `bundle_manifest.json` 은 `signature.key_id` 필드만 보유하고 로테이션 절차는 미정의. Edge 는 `BundleStore._verify_signature(pubkey)` 에 공개키를 런타임 주입 받는 구조 (R8 구현).

**Edge 권고 — `bundle-signing.md` 초안**:

1. **저장 위치**: Edge 는 `TRUSTED_PUBKEYS` env (colon-separated hex) 또는 `bundle_store/trusted_keys.json` 에서 복수 키 로드. 복수 키 지원이 핵심 — 로테이션 기간에 구키·신키 동시 신뢰.
2. **형식**: ed25519 raw 32-byte public key, hex 인코딩. `key_id` = SHA256(pubkey_raw) 앞 16자리.
3. **로테이션 플로우**:
   - Day 0: GB 신규 키 생성 → Edge 에 신키 배포 (env 업데이트)
   - Day 7: GB 가 구키·신키 모두로 이중 서명 또는 신키로 전환
   - Day 30: 구키 제거
4. **검증**: Edge 는 `manifest.signature.key_id` 를 trusted_keys 에서 검색 → 없으면 번들 거부.
5. **긴급 폐기**: 구키 compromise 시 `TRUSTED_PUBKEYS` 에서 제거 → 해당 키로 서명된 번들 즉시 활성화 불가.

GB 가 이 초안을 `energy-contracts/protocols/bundle-signing.md` 신설로 수용하면 Edge 가 구현 (다음 Phase).

**교차 로직** (Edge 가 즉시 구현 가능):

```python
def recommended_techs(capability, manifest):
    out = []
    for tech in manifest["tech_list"]:
        if capability["backend"] not in tech.get("supported_backends", []):
            continue
        btypes = tech.get("applicable_building_types", [])
        if btypes and capability["building_type"] not in btypes:
            continue
        out.append(tech["id"])
    return out
```
